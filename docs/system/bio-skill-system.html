Bio Skill System V1 Architecture

这一页把 v1 生物信息 skill 系统的控制层讲清楚。目标不是先把所有生信 workflow 做完，而是先把自然语言需求、候选 plan、用户确认、skill 路由、阶段执行、暂停恢复这条主链跑通。

核心原则是 plan-first orchestration。用户先确认 plan， agent 再执行 plan 的展开版，而不是拿到需求后临场 improvisation。

系统中心 自然语言需求先进入调度层，变成多个可比对的 plan，之后才进入实际 skill 执行。
v1 边界 先打透通用调度框架，不先绑定 RNA-seq、结构预测或数据库检索中的某一个。
运行方式 本地 skill 仓为能力底座，agent 提示编排负责控制流，后续再决定是否抽象成 MCP 层。
默认执行策略 半自动。每个阶段结束都可汇报，遇到高风险、外部依赖、大计算或覆盖写入时停下确认。

核心对象

Request 用户的自然语言需求和上下文。它是归一化前的意图入口，也是整个系统的输入合同。
Plan 由系统生成的候选方案。用户比较、修改、选择、确认的对象。确认后，它变成执行合同。
Skill 最小能力单元。它可以是单工具、工作流模板、控制器行为或审查逻辑，但边界必须清晰。
Workflow 由多个 skill 组成的可复用组合。它承接常见分析路径，但不代替 orchestrator。
Run 一次被批准 plan 的具体执行实例。它记录阶段状态、人工决策、产物、问题和恢复点。

请求到执行流程

用户真正控制的是 plan，不是命令。系统真正执行的是已批准 plan 的阶段展开版。

1. Request 用户提交自然语言需求
->
2. Normalize 提取输入、目标、约束、缺口
->
3. Generate Plans 生成至少两个候选方案
4. User Review 选择、删减、重排或改写 plan
->
5. Approve plan 进入 approved 状态
->
6. Expand Draft 生成分阶段执行清单
7. Route Skills 每阶段映射候选 skill 集
->
8. Execute Stage 半自动执行并产出 artifact
->
9. Validate + Save Run 校验产物并记录恢复点
10. Resume or Deliver 继续下一阶段，或中断后恢复
至少两个候选 plan: conservative, efficient
可选第三个: exploratory
默认每阶段结束就可汇报

Skill 分类

这一步是把“很多 skill”变成“可调度系统”。v1 不先修全量，而是先形成可用核心集。

bio-skill-system
orchestrator
  • 负责 plan、路由、执行控制和恢复
  • 面向系统行为，不面向单一工具
workflow
  • 封装多步组合任务
  • 复用高频分析骨架
atomic
  • 单工具、单动作、单边界
  • 输入输出和副作用必须明确
policy
  • 环境检查、输入审计、产物验证
  • 给执行加安全和质量约束
当前仓库中的大量 CLI help 型 skill 应优先归为 atomic，不应该继续平铺在系统最外层。

Plan 与 Run 的关系

`Plan` 是用户确认的执行合同，`Run` 是执行后的轨迹。它们不能混在一起。

Plan
  • plan_id, request_id, title, summary
  • strategy_type, assumptions, prerequisites
  • expected_outputs, risks, estimated_cost
  • stages: goal, candidate_skills, validation
  • approval_state: draft / selected / approved
Run
  • run_id, plan_id, request_id, status
  • current_stage, stage_status
  • stage_order, stage_context
  • validation_results by stage
  • artifacts and artifact paths
  • decisions and issues
  • resume_from, last_update
Plan Approved 执行合同被冻结
->
Run Created 产生具体执行实例
->
Pause / Resume 通过 run state 恢复，而不是重想 plan

run-resumer 从已有 run state 恢复执行 run object resume context

半自动执行循环

半自动不是“总要人工点下一步”，而是“默认继续，但在关键节点停下”。执行控制应该是状态机，不是聊天里的临时判断。

Stage Ready prerequisites 满足
->
Risk Check 下载 / 覆盖 / 大计算 / 外部依赖
->
Execute 调 workflow 或 atomic skills
Validate 检查 artifact 与 stage validation
->
Save Run 写入状态、产物、决策、问题
->
Report 汇报本阶段结果与下一步
Continue 进入下一阶段
or
Pause 等用户确认
or
Resume Later 从 run state 恢复

当前可运行链路

v1 现在已经有一条能实际跑通的 CLI 主链，不只是概念图。当前重点不是把所有生信 workflow 做满，而是先把 plan 审批后如何进入 run state、如何按阶段推进、如何暂停恢复这条骨架稳定下来。

Planning propose -> review -> approve
->
Adjust edit 可选，用于改风险与确认点
->
Expand draft 生成阶段执行草稿
Run Init run-init 冻结第一份 run state
->
Advance run-next-stage 推进一个 stage
->
Hold / Resume run-pause 或 run-resume
->
Inspect run-status 看当前暂停点和下一动作
->
Review run-review 看阻塞点、未来风险和是否可继续
confirmation 阶段默认先进入 paused，显式 --confirm 后再执行。
每次 stage 推进会写入 validation_results、artifacts 和 resume_from。
run-status 输出当前 stage、进度分组、最近问题和建议下一动作。
run-review 输出当前 run 的 verdict、blocking issues 和 future issues。
当前本地验证已覆盖 `pytest` 回归与 RNA-seq smoke run。

V1 交付物

必须先有 request / plan / run schema，skill registry，workflow registry，routing rules。
必须先能解释 为什么选这个 plan、为什么走这组 skill、为什么在这里暂停。
必须先能恢复 任意阶段中断后，系统都能从 run state 恢复，而不是重新生成一遍。
