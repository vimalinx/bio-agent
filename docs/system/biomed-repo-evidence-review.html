高权重证据
  • 真实目录统计：`SKILL.md`、`.py`、`test_*.py`、`memory/workflows/*.yaml`。

  • 有 SKILL.md 不等于有 runtime；有 Python 文件也不等于能跑通服务。

推荐拼装方案：怎么把它们变成你要的有机 skill 系统

目标不是“选一个仓库全盘照搬”，而是把不同仓库里真正有价值的层拆出来重组。

1. Control Plane 以 bio-agent 为底
  • 保留 request 归一化、候选 plan 生成、用户批准、阶段执行的骨架。
  • 把缺失的 bridge 层补齐，不要再让 `project.py` / `litefold.py` 导入即崩。
  • 把 policy 层从概念变成真实可运行节点。
2. Domain Backend 借 OpenBioMed
  • 把 OpenBioMed 当工具/模型后端，而不是总调度器。
  • 优先接入它真实存在的 tool registry、inference pipeline、数据对象模型。
  • 把 broken workflow/report 服务视作待修模块，不要直接托底依赖。
3. 工程化借 ClawBio，知识面借 LabClaw
  • 从 ClawBio 抄复现包、skill 目录规范、CI 与测试策略。
  • 从 LabClaw 和 OpenBioMed skills 提取高层任务模板、skill 路由语义和领域覆盖。
  • 不要把文档 skill 当成执行能力；执行面一定要落到真实 backend。
