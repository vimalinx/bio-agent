Bio-Agent / ClawBio / LabClaw / OpenBioMed 代码真实性审计

这页不是对 README 的复述，而是对四个本地仓库做“真代码优先”的审计稿。 结论只分三类：代码直接支持、代码部分支持、文档声称但本地状态未证实。 你要做的是一个“用户自然语言提出需求，AI 先产出 plan，用户选择或修改后再执行”的通用调度框架； 因此本页重点判断每个仓库更适合落在 control plane、skill layer、execution backend 还是 knowledge layer。

bio-agent 真实强项是 plan / propose / approve / run 这条控制面骨架。 真正像“总调度器”，但桥接脚本和外围工程还没完全收口。
ClawBio 真实强项是可执行 skill 产品化、CI、通用报告/校验件和本地数据处理。 更像可落地的 skill 产品库，而不是总调度框架。
LabClaw 真实强项是 breadth，不是 execution。 它本地几乎就是一个大型 SKILL.md 知识层。
OpenBioMed 真实强项是 biomedical toolkit + model/tool pipeline。 更适合做执行后端，不适合直接当你的通用主控框架。

推荐融合架构：四仓不该互相替代，而该分层组合

Layer 1: 用户交互与计划确认
Web UI / Chat UI。展示候选 plan、允许用户选择、编辑、批准、暂停、恢复。
Layer 2: Control Plane
用 bio-agent 的 request normalization、workflow matching、 candidate plan generation、approval state 作为主控制面。
Layer 3: Skill Routing + Knowledge
让 ClawBio 和 OpenBioMed 的真实可执行能力进入统一 skill registry； 让 LabClaw 主要提供领域提示、术语路由、工具说明和长尾任务模板。
Layer 4: Execution Backend
以 OpenBioMed 承担模型/分子/蛋白/单细胞执行， 以 ClawBio 承担面向具体技能的产品化执行和报告输出， 必要时再桥接系统工具、Galaxy 或外部科学软件。
Layer 5: Provenance / Report / Replay
借鉴 ClawBio 的 checksum + result.json + report.md 思路， 给每次 agent 执行保留输入摘要、命令、环境、关键中间件和可复跑证据。

关键原则 不要让一个仓库同时承担“计划审批 + 知识层 + 执行后端 + 报告可复现”全部职责。 这四块在当前现实代码里分散在不同仓库上，强行一把梭会把系统做成又重又不稳。

你要的主流程，应该长这样

下面这个流程是基于四仓真实长板拼出来的，不是抄任何一个仓库现状。

1. 用户需求 自然语言输入
→
2. 请求规范化 tags / constraints / inputs
→
3. 产出候选 plans conservative / efficient / domain-specific
→
4. 用户选择或编辑 审批、删改、补充约束
→
5. 编译成执行图 stage → skill → backend
→
6. 执行与检查点 中间结果、失败重试、人工插入
→
7. 产出报告 结果、证据、复跑包
应该来自 bio-agent 的部分 request schema、routing rules、candidate plans、approval state、run state。
应该来自 ClawBio 的部分 统一报告结构、checksum、result envelope、测试习惯、面向具体技能的可执行模板。
应该来自 LabClaw 的部分 长尾技能知识、术语表、研究任务提示模板，不直接负责执行。
应该来自 OpenBioMed 的部分 分子/蛋白/单细胞/数据库工具与模型执行后端。

下一步建议

  1. 先把 bio-agent 固化成统一控制面协议： Request → CandidatePlans → ApprovedPlan → ExecutionGraph → RunTrace → EvidenceBundle。
  2. 给 ClawBio 和 OpenBioMed 各写一层 adapter， 统一输入输出，而不是让上层 UI 直接调用它们内部对象。
  3. 把 LabClaw 转成可检索知识库或 routing hints，不要把它误当执行引擎。
  4. 引入统一 provenance 规范：至少要有 report.md、 result.json、输入摘要、校验和、关键命令或 workflow dump。
  5. 在真正做大规模集成前，先挑两个高价值路径打样： RNA-seq 和 target-based lead design。 这两条一条偏传统生信，一条偏药物发现，足够把控制面与执行层打通。
