OpenBioMed

一句话定位 真实定位是 biomedical toolkit / inference platform first，skills second。
  • 本地统计：45 个 skills、36 个 tools、118 个 open_biomed/ Python 文件、2 个测试文件、1 个内置 memory workflow。
  • 核心层真实存在： agent.py、 workflow.py、 pipeline.py、 tool_registry.py、 run_server.py。
  • 它可以把 agent 轨迹导出成 workflow，见 agent.py 第 770 行后的 export_as_workflow。
必须降级的地方
  • workflow server 当前导入就坏： run_server_workflow.py 第 11 行导入不存在的类。
  • setup.py 第 7 行把 install_requires 留空，但 requirements.txt 里有 RDKit、FastAPI、LangGraph、Scanpy 等关键依赖。
  • python -m pytest -q tests 在本地暴露出 rdkit 缺失；普通 pytest -q tests 还会先遇到 open_biomed 导入路径问题。
  • agent.py 第 151 行使用 create_subprocess_shell， 第 199 行有可变默认参数， 第 180 行和 228 行的 docker 执行函数仍是 pass。
代码支持的强项
  • 不是纯 skills repo，是真实代码量较厚的平台仓。
  • 工具注册表和推理脚本之间绑定紧密，skills 层并非完全空壳。
  • 在生物医药模型、数据对象和 tool backend 上，比前面三个仓都更厚。
对你的系统的相关性
  • 相关性很高：执行后端。
  • 不建议直接当总调度框架，但非常适合接在你的 control plane 后面，充当 biomedical execution engine。
  • 特别适合作为 tool backend、inference backend 和高价值 skill 的实际运行层。

建议的整合架构

你的目标不是简单拼仓库，而是把四个仓拆成不同层次的资产。下面这个分层，比“选一个仓全盘继承”更贴近真实代码能力边界。

Control Plane 用户确认、计划版本、暂停恢复、阶段审批
首选：bio-agent
吸收点：request → propose → review → approve → draft → run
必须补：缺失模块、测试隔离、真实 policy 层
Skill Layer 任务模板、提示编排、领域路由
首选素材：LabClaw + OpenBioMed skills
吸收点：任务意图词、技能说明、分析套路
必须补：把文档 skill 映射到真实可执行 backend
Execution Backend 模型推理、数据对象、工具执行
首选：OpenBioMed
补充：ClawBio 的成熟技能实现
必须补：依赖声明、服务修复、执行安全
Evidence / Repro 结果包、fixture、可复现输出、回归测试
首选：ClawBio
吸收点：fixture 测试、输出资产、repo-level pytest discipline
必须补：统一到你的主控协议与 run artifact 规范

直接结论：主框架用 bio-agent 的 control plane，后端接 OpenBioMed，执行质量借鉴 ClawBio，提示与知识面吸收 LabClaw。 这是最符合真实代码现状、也最不容易被 README 吹法误导的组合。
